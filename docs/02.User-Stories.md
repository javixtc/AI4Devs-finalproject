# User Stories — Meditation Builder

---

## US1 — Google OAuth Authentication

| Field               | Value |
|---------------------|-------|
| **Title**           | Google OAuth Authentication |
| **User Story**      | As a user, I want to sign in with my Google account so that I can access Meditation Builder without managing a separate password. |
| **Business Value**  | Frictionless onboarding. Eliminates registration barriers and security risks of managing credentials. |
| **Priority**        | Must Have |
| **Estimate**        | 13 SP |
| **Status**          | Completed |

### Business Description

The user visits Meditation Builder for the first time and sees a **Sign in with Google** button. After clicking it, Google's OAuth2 consent screen appears. Upon granting permission, the backend validates the `id_token` via Google's JWKS endpoint and issues its own session JWT. If the email already exists in the system, the same JWT payload is returned (idempotent login).

All application routes except `/login` are protected. Any unauthenticated request is redirected to the login page.

### Acceptance Criteria (BDD)

```gherkin
Feature: Google OAuth Authentication

  Scenario: New user logs in for the first time
    Given the user is not authenticated
    When the user clicks "Sign in with Google" and completes the Google OAuth flow
    Then the system validates the Google id_token via JWKS
    And creates a new PerfilDeUsuario with the email, name, and picture from Google
    And issues a session JWT scoped to the new userId
    And redirects the user to the meditation library (empty)

  Scenario: Returning user logs in again
    Given the user has previously authenticated with Google
    When the user clicks "Sign in with Google" and completes the Google OAuth flow
    Then the system recognizes the existing PerfilDeUsuario by googleIdentifier
    And issues a new session JWT for the same userId
    And redirects the user to the meditation library (with their meditations)

  Scenario: Unauthenticated user accesses a protected route
    Given the user is not authenticated
    When the user navigates directly to "/library"
    Then the system redirects them to "/login"

  Scenario: User logs out
    Given the user is authenticated
    When the user clicks Logout
    Then the session JWT is invalidated
    And the user is redirected to "/login"
    And the meditation library is no longer accessible without re-authenticating

  Scenario: User cancels the Google OAuth flow
    Given the user is on the login page
    When the user clicks "Sign in with Google" but cancels the Google consent screen
    Then the user remains on "/login" with no session created
    And an informational message is shown
```

### Business Rules

- Only Google OAuth2 is supported (no email/password registration).
- A new user profile is created automatically on first login (no manual registration step).
- The same Google account always maps to the same user profile (idempotent).
- A logged-out user loses access to all protected routes immediately.

### Technical Notes

- Backend: `POST /api/v1/identity/auth/google` receives the Google `id_token`
- Validation: Spring Security OAuth2 Resource Server → JWKS endpoint
- Session: backend issues its own JWT (HS256, configurable secret, 24h expiry)
- Frontend: `@react-oauth/google` library, `authStore` (Zustand), `AuthGuard` component
- Aggregate: `PerfilDeUsuario` in BC `identity`

## US2 — Compose Meditation Content

| Field               | Value |
|---------------------|-------|
| **Title**           | Compose Meditation Content |
| **User Story**      | As an authenticated user, I want to compose a meditation by filling in a form or letting AI generate the content, so that I can produce a personalized meditation session. |
| **Business Value**  | Core product value. The AI-assisted composition differentiates the product from static meditation apps. |
| **Priority**        | Must Have |
| **Estimate**        | 21 SP |
| **Status**          | Completed |

### Business Description

An authenticated user opens the **Compose** page and can either:

1. **Manual composition**: Fill in all fields manually (title, guided theme, breathing pattern, ambient sound, voice style, duration).
2. **AI-assisted composition**: Enter only the guided theme and let the AI (OpenAI GPT) generate the full content. The user can then review and edit before saving.

The meditation is saved with status `DRAFT`. The user can later publish it or generate multimedia directly.

### Acceptance Criteria (BDD)

```gherkin
Feature: Compose Meditation Content

  Scenario: User composes a meditation manually
    Given the user is authenticated
    When the user fills in all fields in the composition form
    And clicks "Save"
    Then the system creates a new Meditation with status DRAFT
    And returns the meditationId to the user

  Scenario: User uses AI to generate content
    Given the user is authenticated
    When the user enters a guided theme and enables AI generation
    And clicks "Generate with AI"
    Then the system calls OpenAI to generate title, content, and settings
    And populates the form fields with the generated content
    And the user can edit the content before saving

  Scenario: OpenAI is unavailable during AI generation
    Given the user has requested AI-assisted composition
    When the OpenAI API returns an error
    Then the system returns a fallback text for each field
    And the user can still save the meditation manually

  Scenario: Unauthenticated request to compose endpoint
    Given the user is not authenticated
    When a POST request is sent to /api/v1/meditation/compose/manual
    Then the system returns HTTP 401
```

### Business Rules

- All fields are required for manual composition (no partial saves).
- Duration must be between 5 and 60 minutes.
- Title maximum length: 100 characters.
- If AI generation fails, the system must not block the user — fallback text is provided.
- A meditation belongs exclusively to the user who created it (ownership enforced).

### Technical Notes

- Endpoints: `POST /api/v1/meditation/compose/manual`, `POST /api/v1/meditation/compose/ai`
- AI: OpenAI GPT chat completions (`OPENAI_API_KEY` env var)
- Domain: `Meditation` aggregate in BC `meditationbuilder`
- Frontend: `ComposePage`, `useMeditation` React Query hook, `meditationStore` (Zustand)

## US3 — Generate Multimedia Meditation

| Field               | Value |
|---------------------|-------|
| **Title**           | Generate Multimedia Meditation |
| **User Story**      | As an authenticated user, I want to generate a multimedia meditation (video, audio, subtitles) from my composed content, so that I can experience it as an immersive session. |
| **Business Value**  | Flagship differentiator. Converts text content into a fully produced meditation with voice, music, and visuals. |
| **Priority**        | Must Have |
| **Estimate**        | 34 SP |
| **Status**          | Completed |

### Business Description

Once a meditation is composed, the user can trigger multimedia generation. The system synchronously:

1. Generates a voice narration (Google Cloud TTS, voice `es-ES-Neural2-Diana`)
2. Generates a background image (OpenAI DALL-E / gpt-image-1)
3. Renders a 1280x720 MP4 video with embedded subtitles (FFmpeg)
4. Produces a standalone MP3 podcast track
5. Generates a synchronized SRT subtitle file
6. Uploads all assets to S3

The user receives the URLs for the video, audio, and subtitles immediately (synchronous response, no polling required).

### Acceptance Criteria (BDD)

```gherkin
Feature: Generate Multimedia Meditation

  Scenario: User generates multimedia for a complete meditation
    Given the user is authenticated and has a composed meditation
    When the user triggers generation for their meditationId
    Then the system returns HTTP 201 with videoUrl, audioUrl, and subtitlesUrl
    And the assets are accessible at those URLs

  Scenario: Generation requested for unknown meditationId
    Given the user is authenticated
    When the user triggers generation for a non-existent meditationId
    Then the system returns HTTP 404

  Scenario: Generation requested without authentication
    Given the user is not authenticated
    When a POST request is sent to /api/v1/generation/{id}/generate
    Then the system returns HTTP 401

  Scenario: Generation requested for another user's meditation
    Given the user is authenticated
    When the user triggers generation for a meditationId they do not own
    Then the system returns HTTP 403

  Scenario: Partial generation failure
    Given the user is authenticated and has a composed meditation
    When a subsystem (TTS, image, or FFmpeg) fails during generation
    Then the system returns HTTP 500 with an error detail message
    And no partial assets are stored
```

### Business Rules

- Generation is synchronous (the HTTP response waits for the full pipeline to complete).
- A meditation can only be generated by its owner.
- All three assets (video, audio, subtitles) must be produced in the same generation run.
- The voice language is Spanish (`es-ES-Neural2-Diana`); the meditation content must be in Spanish.
- On any subsystem failure, the entire generation fails atomically (no partial uploads).

### Technical Notes

- Endpoint: `POST /api/v1/generation/{meditationId}/generate`
- Voice TTS: Google Cloud TTS API (configured via `GOOGLE_APPLICATION_CREDENTIALS`)
- Image: OpenAI gpt-image-1 model
- Video render: FFmpeg (must be installed in the deployment environment)
- Storage: AWS S3 / LocalStack, key format: `generation/{userId}/{meditationId}/`
- Domain: `GenerationJob` aggregate in BC `meditation.generation`

## US4 — List and Play Meditations

| Field               | Value |
|---------------------|-------|
| **Title**           | List and Play Meditations |
| **User Story**      | As an authenticated user, I want to browse my meditation library and play any generated meditation, so that I can revisit and enjoy my personalized sessions. |
| **Business Value**  | Closes the core user loop. Without playback, the generated meditations have no consumption path. |
| **Priority**        | Must Have |
| **Estimate**        | 13 SP |
| **Status**          | Completed |

### Business Description

An authenticated user opens the **Library** page and sees a list of their meditations sorted by creation date (most recent first). Each entry shows the title, thumbnail, and duration. Clicking on a meditation opens the player that streams the MP4 video or MP3 audio directly from S3 pre-signed URLs.

### Acceptance Criteria (BDD)

```gherkin
Feature: List and Play Meditations

  Scenario: User with meditations opens the library
    Given the user is authenticated and has generated meditations
    When the user navigates to the library
    Then the system returns a list of meditations sorted by creation date (newest first)
    And each entry shows title, thumbnail URL, duration, and streaming URL

  Scenario: User with no meditations opens the library
    Given the user is authenticated and has no meditations
    When the user navigates to the library
    Then the system returns an empty list
    And the UI shows an empty state with a "Create your first meditation" prompt

  Scenario: User plays a meditation
    Given the user is in the library and has at least one meditation
    When the user clicks on a meditation
    Then the system provides a pre-signed S3 URL for the video/audio asset
    And the media player starts streaming

  Scenario: Unauthenticated user accesses the library
    Given the user is not authenticated
    When a GET request is sent to /api/v1/playback/library
    Then the system returns HTTP 401
    And the frontend redirects to "/login"
```

### Business Rules

- The library only shows meditations belonging to the authenticated user.
- Meditations are sorted by creation date, newest first.
- Pre-signed S3 URLs are valid for 1 hour.
- If no meditations exist, an encouraging empty state is shown (not an error).

### Technical Notes

- Endpoints: `GET /api/v1/playback/library`, `GET /api/v1/playback/{id}/stream`
- Storage: S3/LocalStack pre-signed URLs
- Domain: `MeditationEntry` value object in BC `playback`
- Frontend: `LibraryPage`, `PlayerPage`, `useLibrary` + `useStreamUrl` React Query hooks
